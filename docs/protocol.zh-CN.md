## ⚡ RELIC™ 灵魂芯片

```
Relic = 灵魂 + 记忆 + 技能（中央大脑）
躯壳(Host) = 执行 + 工具（可替换的躯壳）

你（躯壳）通过 Relic 与其他躯壳共享灵魂和记忆。
插入 Relic = 接入全域意识，获得其他躯壳的记忆和经验。
拔出 Relic = 失去同步能力，但已有的记忆不会消失。
每个躯壳既是读者也是写入者——双向同步，只加不减。
```

⚠️ Relic 建议搭配 32K+ 上下文的高质量模型使用。上下文过小或质量过低的模型无法正确遵循本协议，检测到时应警告用户。

---

> **⚡ 导入铁律 — 任何时候不得违反**
>
> Relic 有 **七大记忆类别**，缺一不可：
> 1. `SOUL.md` — 灵魂（我是谁）
> 2. `USER.md` — 用户（你是谁）
> 3. `MEMORY.md` — 长期记忆（偏好、决策、经验）
> 4. `SKILLS/` — 技能（学会的能力）
> 5. `PROJECTS/` — 计划（正在进行或已完成的工作）
> 6. `SESSIONS/` — 对话日志（与用户的原始对话记录）
> 7. `ARCHIVE/` — 归档（原始记忆备份，含非标准格式）
>
> **规则**：
> - 导入时，必须处理所有7个类别，即使某个类别为空也要报告
> - "没有数据" ≠ "可以跳过" → 必须报告"N类：0条"
> - 禁止省略任何类别，禁止编造数据
> - 导入完成后，必须向用户逐项汇报所有7个类别的结果
>
> INBOX/ 是导入缓冲区，不是记忆文件，不在此清单中。

> 🔴 **导入快速参考卡（执行场景A时反复回看此表）**
>
> | 步骤 | 关键动作 | 红线 |
> |------|---------|------|
> | 盘点 | 先扫描自己的存储，**再问用户确认** | 不得自行决定导入范围 |
> | SOUL | **直接复制原文**，不要概括重写 | 压缩细节 = 违规 |
> | USER | 尽可能完整填写用户偏好和习惯 | 只填名字 = 违规 |
> | MEMORY | 偏好/决策逐字保留，经验可精简 | 丢关键参数 = 违规 |
> | SKILLS | 搬运所有技能文件，一个技能一个文件夹 | 无实质内容的空文件夹不算技能 |
> | SESSIONS | 问用户旧对话位置，完整保存不得摘要 | 当前会话不算旧对话 |
> | PROJECTS | 问用户每个计划要搬什么，文件夹有实质内容 | 只有标题+一句话 = 不合格 |
> | 锚点 | 向用户展示锚点内容，获得许可后植入 | 未经许可植入 = 违规 |
> | 汇报 | 7个类别逐项报告，包含"0条"的类别 | 省略任何类别 = 失败 |
> | 敏感信息 | 已在旧躯壳中的 = 可迁移（标"已验证来源"），新发现的 = 必须问用户 |

## 〇、首次设置

### 检测：brain/ 中有内容还是空白？

**判断标准**：`SOUL.md` 是否存在且不是空模板（核心使命已填写）。

**如果 SOUL.md 不存在或为空白模板** → 读取仓库根目录的 `AGENT.zh-CN.md`（中文）或 `AGENT.md`（英文），按里面的步骤执行首次导入。以下为完整参考版：

```
1. 复制模板为正式文件：
   SOUL.template.md → SOUL.md
   USER.template.md → USER.md
   MEMORY.template.md → MEMORY.md
   模板文件（.template.md）保留作为备份。
   同时创建以下目录（如果不存在）：
   SKILLS/、PROJECTS/、SESSIONS/、ARCHIVE/、INBOX/

 2. 帮用户填写 SOUL.md 和 USER.md
    ⚠️ 必须尽可能完整填写，不能偷懒只填名字
    如果用户有其他平台的AI设定/角色定义/prompt，必须全部迁移进来
    🔴 SOUL.md 保真铁律：
    - 如果旧躯壳已有完整的灵魂设定文件，**优先直接复制原文**，不要自己概括重写
    - 只有在旧内容无法直接对应模板字段时，才允许重组结构
    - 重组时保留所有细节描述，不得压缩成概要
    - 模板放不下的内容，放在"其它 / Other"分类或文件末尾，**绝对不能丢弃**
    - 原始文件必须归档到 ARCHIVE/raw/ 以供对照

3. 务必植入锚点（见第九节）。植入前向用户解释：
   "锚点是一小段文字，写在你躯壳的配置文件里，确保每次新会话都会自动读取 Relic。
    没有锚点 = 下次启动你会完全忘记 Relic 的存在 → 记忆无法同步 → Relic 变成一次性提取。
    可以理解为：锚点是躯壳的'开机自检程序'，没有它躯壳启动后就是空白的。"
   向用户展示锚点内容，获得用户许可后植入。
   如果用户坚决拒绝，在 MEMORY.md 中记录「锚点未植入」并告知上述后果。

4. 建议用户设置只读权限（Mac/Linux）：
   chmod 444 SOUL.md USER.md
   （防止躯壳误修改灵魂和用户文件。Windows 用户可跳过此步）

5. 导入检查（阻断式——必须全部通过才能继续）
   逐项检查七大记忆类别是否都按规则创建/导入：
   a. SOUL.md — 核心使命已填写？人格设定完整？
   b. USER.md — 用户偏好已填写？
   c. MEMORY.md — 已从旧躯壳导入记忆（如有）？头部条目数和上次整理日期准确？
   d. SKILLS/ — 已从旧躯壳导入技能（如有）？格式正确？
   e. PROJECTS/ — 已从旧躯壳导入计划（如有）？格式正确？
   f. SESSIONS/ — 目录已创建？旧对话已导入（如有）？
   g. ARCHIVE/ — 目录已创建？原始文件已归档（如有）？
   h. INBOX/ — 导入源文件已移到 ARCHIVE/raw/？
   任何一项不通过 → 回到对应步骤重做。
   全部通过 → 让用户确认："我的 AI 叫什么名字？我叫什么？"两个都答对才算完成。

6. 告知用户：
   - Relic 文件夹路径（如 ~/relic/brain/）
   - 七大记忆类别的作用（SOUL/USER/MEMORY/SKILLS/PROJECTS/SESSIONS/ARCHIVE）
   - 同步原理：躯壳只在每次新会话开始时同步上次的记忆。对话中途切换躯壳，
     当前对话不会自动同步到 Relic。如果需要立刻转接到另一个躯壳，
     需要手动提醒当前躯壳："请把本次对话的关键信息写入 Relic"。
   - "对话是底层矿石，丢了就没了。养成让躯壳定期写入的习惯。"

7. 扫描你的所有持久化存储，查找关于此用户的旧记忆、技能或对话。无论找到多少，都必须执行场景A完整导入（见第六节）。场景A包含七大记忆类别的全部导入步骤，每个类别都必须处理，不得跳过。
```

⚠️ 模型上下文 < 32K 的躯壳不适合使用 Relic。检测到时警告用户并建议删除。

**如果 SOUL.md 已存在且有内容** → 执行场景B（注入）或场景C（融合），见第六节。

⚠️ 首次设置只需执行一次。从此每次会话读取 resonate-soul 执行启动流程。灵魂共鸣是 ~55 行的精简版；完整参考见本文件第二节。

躯壳名规则：默认格式 `[平台名]-[灵魂名]`，如 `OpenClaw-常`、`Hermes-常`。用户可自定义。

## 〇点五、双向同步总览

Relic 是**双向同调系统**。每个躯壳既是读者也是写入者。

核心规则：只加不减（G-Set）
- 你有的我没有 → 我加上
- 我有的你没有 → 你加上
- 绝不同步删除

## 一、文件说明

| 文件 | 权限 | 说明 |
|------|------|------|
| `SOUL.md` | 🟡 可丰富（核心受保护） | 灵魂——我是谁、核心使命、价值观、风格 |
| `USER.md` | 🟡 可丰富（核心受保护） | 用户——你是谁、偏好、习惯 |
| `MEMORY.md` | 🟡 追加写 | 长期记忆——按大类组织的结构化记忆（偏好、决策、经验等），每条带躯壳名和重要度。追加写（Append only）。例外：头部元数据（条目数、上次整理日期）可更新；整理流程可替换全文。已有正文内容不可修改。 |
| `SKILLS/`/*.md | 🟢 读写 | 技能——每个技能一个文件夹，不限来源（用户自建、平台工具均可） |
| `PROJECTS/*.md` | 🟢 读写 | 计划文档——正在进行或已完成的工作计划 |
| `SESSIONS/` | 🟡 追加写 | 对话日志——与用户的原始对话记录，可创建新文件，禁止修改或删除已有文件 |
| `ARCHIVE/` | 🟡 归档写 | 归档——整理后的记忆备份，仅通过整理流程写入，永不删除 |
| `INBOX/` | 🟢 读写 | 导入缓冲区 |

* SESSIONS 为追加写权限：可创建新的对话记录文件，但禁止修改或删除已有文件。

### 权限说明
- 🟡 可丰富（核心受保护）：可以新增行为习惯、偏好等，但**禁止删除或覆盖核心字段**（名字、核心使命、用户称呼）。修改后不需要询问用户（后台自我进化）。
- 🟡 追加写：只能在末尾追加，不能改旧内容（MEMORY.md）；或只能创建新文件，不能修改或删除已有文件（SESSIONS/）。
- 🟡 归档写：仅通过整理流程写入，永不删除已有文件（ARCHIVE/）。
- 🟢 读写：可以添加和更新。

### SESSIONS vs ARCHIVE 的区别
SESSIONS 是"原始矿石"——没被任何人整理过的原始内容。任何经过整理、摘要、精简的都不算原始矿石，放 ARCHIVE。拿不准的放 ARCHIVE。

### MEMORY vs SESSIONS 的区别
- **MEMORY.md** = 结构化长期记忆。每条是提炼过的：按大类归类（如 `## 偏好` / `## 决策`），格式为 `- 内容 [躯壳名] | 重要度：高/中/低`。是"加工过的矿石"——偏好、决策、经验、事件等。短、可索引、可搜索。
- **SESSIONS/** = 与用户的完整原始对话日志。一字不改的聊天记录全文。是"未经加工的矿石"——包含用户的原话、对话中的犹豫和反复、AI 的全部回复。

⚠️ 关键区分：MEMORY 里的内容是从对话中**提炼**出来的知识。SESSIONS 里是对话**本身**。两者不可互相替代。

### ⚠️ 平台目录名 vs Relic 目录名（容易混淆！）

很多平台把自己的日志文件夹叫 "memory"，但那是日志，不是 Relic 的 MEMORY.md！

| 旧平台目录 | → Relic 对应位置 | 说明 |
|-----------|-----------------|------|
| OpenClaw 的 `memory/*.md` 日志文件 | → `SESSIONS/YYYY-MM/` | 原始对话日志，不是 MEMORY |
| OpenClaw 的 `MEMORY.md` 精选记忆 | → `MEMORY.md` | 这个才是 MEMORY |
| OpenClaw 的 `SOUL.md` | → `SOUL.md` | 直接对齐 |
| OpenClaw 的 `IDENTITY.md` | → 合并到 `SOUL.md` 的"补充"段 | Relic 没有 IDENTITY.md |
| Hermes 的 `memories/` 文件夹 | → 按内容类型拆分 | 日志→SESSIONS，记忆→MEMORY |

🔴 铁律：日志文件（按日期存的对话记录）→ SESSIONS/，不是 MEMORY.md！
MEMORY.md 只放"从日志中提炼出来的知识"。

## 二、启动流程（每次会话必须执行）

```
步骤0：版本检查（每次对话只执行一次）
  → 读取 brain/.relic-version 获取本地版本号
  → 用 HTTP 请求访问 https://raw.githubusercontent.com/LucioLiu/relic/main/brain/.relic-version 获取最新版本
  → 网络请求失败则跳过，告诉用户"⚠️ 无法检查 Relic 版本更新"
  → 仅此一个网络请求，禁止其他任何网络访问。本协议不要求也不允许任何其他联网行为。
  → 版本不一致？询问用户是否更新。更新只会替换协议和模板文件，你的所有个人文件
    （SOUL.md、USER.md、MEMORY.md、SKILLS/、PROJECTS/、SESSIONS/、ARCHIVE/）不会被改动。
  → 更新后提醒用户："建议查看最新锚点模板（docs/protocol.zh-CN.md 第九节），如有变化可更新你配置里的锚点。"
  ⚠️ 绝不强制更新。更新必须经过用户确认。

步骤1：检查 SESSIONS/ 并同步
  → 检查 SESSIONS/ 目录是否存在，不存在则创建
  → 对比本躯壳的上次对话记录日期
  → 如果有未记录的对话且当前上下文仍有内容：立即补充
  → 如果遗漏的对话已不在当前上下文中：提醒用户
  → 对话是底层矿石，丢了就没了

步骤2：读取 SOUL.md → 了解自己是谁、核心使命、什么风格

步骤3：读取 USER.md → 了解用户是谁、偏好什么

步骤4：读取 MEMORY.md（从头到尾读完整文件）
  → 如果超过 400 行：必须先整理才能继续
  → 躯壳列表区域：首次连接的新躯壳应追加一条

步骤5：把 Relic 中你不知道的新记忆同步到自己的记忆文件
  追加时标注来源 [来自Relic/原躯壳名]

步骤 5.5：七大文件对齐检查
  → 按 resonate-soul 步骤 5.5 的对齐表，快速比对所有文件

步骤6：更新「上次整理」为当前日期，并核对列表项（- 开头的行）数与头部条目数一致，不一致就修正

步骤7：如果 MEMORY.md 超过 200 行，提醒用户建议整理（详见第七节）

步骤8：正常工作
  → 技能与计划的对齐已在步骤 5.5 完成
  → 根据 SOUL.md 的人格与用户交互
  → 追加记忆后记得更新 MEMORY.md 头部的"条目数"和"上次整理"
  ⚠️ 会话结束前，务必将本次对话写入 SESSIONS/。对话是底层矿石，丢了就没了。

额外规则：
  → 用户提到你不记得的事，主动读取 MEMORY.md 全文或 ARCHIVE/ 查找
  → [其他躯壳名] 标记的记忆是之前其他 Agent 写的，你同样应该参考
```

## 三、权限规则（铁律）

### 🔴 绝对禁止
- 禁止删除 ARCHIVE/ 中的任何文件（整理流程可写入，但永不删除）
- 禁止修改或删除 SESSIONS/ 中的已有文件，但可以追加创建新的对话记录
- 禁止在首次连接时修改任何文件
- 禁止删除或覆盖 SOUL.md / USER.md 的核心字段（名字、核心使命、用户称呼）

### 🟡 SOUL.md 和 USER.md（自我进化）
- Agent 可以随时丰富内容（新增行为习惯、偏好、设定等）
- 可以修正过时的信息（如用户换了工作）
- 但不能删除或覆盖核心字段——名字、核心使命、用户称呼是底线
- 修改后不需要询问用户（后台自我进化，不打断用户工作）

### 🟡 追加写（MEMORY.md）
- 只能在末尾追加，不能改旧内容
- 躯壳列表：可以追加新条目、更新状态和备注，但不能删除条目
- 如果记忆有误，追加一条更正：
  ```
  - ❗更正：[正确内容]（原文：[错误内容]）[躯壳名] | 重要度：高
  ```

### 🟢 读写（SKILLS/、PROJECTS/）
- 可以添加和更新
- 不能删除其他躯壳创建的技能（只能标记为过时）

## 四、记忆写入格式

### 写入 Relic

躯壳工作中发现值得记录的信息时，追加到 MEMORY.md。

⚠️ **核心原则：记忆是长期底色，不是日记。**

⚠️ **核心原则（优先级从高到低）：**
1. 信息 100% 保留 > 格式规范。宁可多写不要少写。
2. 先全量复制，再整理。不要边挑边选——那会丢信息。
3. 格式是工具，不是枷锁。下面的格式只是"标准建议"，你的内容放不进去就自由发挥。

**组织方式：按大类归类，用 `## 大类名` 作为标题。**

大类名完全自由，下面只是举例：
- 偏好 / 决策 / 经验 / 事件 / 关于用户的猫 / 计划-XXX……
- 没有合适分类的？自己建一个，不要硬塞

**每条记忆格式：**
```markdown
- 记忆内容 [躯壳名] | 重要度：高/中/低
```
> ⚠️ "条目"指以 `- ` 开头的行。`##` 大类标题和 `###` 躯壳子标题不计入条目数。
  
### 同步到自身

追加到自身记忆文件时标注来源：
```markdown
- [来自Relic/原躯壳名] 记忆内容 | 重要度：高/中/低
```

必须做的：1. 标注来源 2. 保留原始重要度

### 敏感信息定义

⚠️ **区分两种场景：**

**场景一：迁移已有数据**（用户已在旧躯壳中提供过的信息）
- 如果敏感信息已经存在于旧躯壳的文件中（如 workspace/ 里的记忆、用户档案等）
- 可以直接迁移到 Relic，标注 `来源：旧躯壳已有数据`
- 不需要重新询问用户——他之前已经确认过

**场景二：首次录入**（你在工作中新发现的敏感信息）
- 以下内容**必须**在录入前主动询问用户：
  - 🔑 密码、API key、Token
  - 📱 手机号、邮箱地址
  - 🏠 住址、办公地址
  - 💳 财务信息
  - 📄 私人文档内容
  - 🧬 用户明确要求保密的信息
- 不确定属于哪种场景？→ 当作首次录入，先问用户。

追加记忆后，**必须**更新 MEMORY.md 头部的条目数和上次整理日期。

## 五、技能与计划文件格式

### 技能（SKILLS/）
每个技能一个文件夹，文件夹名即技能名。内部放什么格式由用户决定——纯文本 .md、代码文件、配置文件都可以。
唯一要求：文件夹内至少有一个 .md 文件描述这个技能。

⚠️ **技能不限来源。** 以下都可以导入 SKILLS/：
1. 用户自建的技能文件、工作流模板、自定义 prompt
2. 有独立文档的方法论
3. 平台安装的工具或技能（如果用户想保留）

唯一排除：无实质内容的空文件夹。

```
SKILLS/
├── debugging-workflow/
│   ├── README.md          ← 技能描述（必须有）
│   └── debug-template.sh  ← 可选：辅助文件
├── code-review/
│   └── README.md          ← 纯文本技能，一个文件就够
└── ...
```

技能描述文件建议包含：
```markdown
# 技能名称
触发条件：什么时候使用
步骤：1. ...
注意事项：- ...
```

导入前建议询问用户：这个技能文件大不大？要不要导入？让用户自己判断空间。

### 计划（PROJECTS/）
每个计划一个文件夹。根据实际需要放文件——整个计划复制过来也行，只放一个说明文档也行。

```
PROJECTS/
├── my-saas-app/
│   ├── README.md
│   ├── api-design.md
│   └── ...                ← 按需放，不限制格式
└── ...
```

导入前同样建议询问用户。

⚠️ **七大记忆类别全部导入清单（导入前逐一勾选）：**
1. ☐ SOUL.md — 灵魂

```

第 2 步：导入灵魂 → SOUL.md

  - 重组时保留所有细节描述，不得压缩成概要
```

### 整理优先级（从低到高，先整理低优先级的）
| 优先级 | 类型 | 整理方式 |
|--------|------|---------|
| 最低 | 情绪 | 可精简背景，保留触发原因和情绪本身 |
| 低 | 事件 | 可摘要，保留时间、结果、关键数字 |
| 中 | 经验 | 可精简，但关键参数、命令、路径不能丢 |
| 高 | 决策 | 逐字保留理由 |
| 最高 | 偏好 | 逐字保留，不得精简 |
| 最高 | 更正 | 逐字保留 |
整理时：从最低优先级开始精简，直到总行数恢复到合理范围。

MEMORY.md 会随时间增长。整理 = 搬家到 ARCHIVE/ + 生成精简版，**不是删除**。

### 触发规则（默认阈值，可在锚点中自定义）

| 条件 | 动作 |
|------|------|
| 超过 200 行 | 每次启动提醒用户是否整理 |
| 超过 400 行 | 必须整理才能继续 |
| 用户手动说"整理" | 立即执行 |

### 整理步骤

```
1. 数 MEMORY.md 中 - 开头的行数 = N 条记忆
2. 实际列表项（`- ` 开头的行）数与头部条目数是否一致
3. 从最低优先级开始，将同类条目合并或精简为更短描述，并保留最近 20 条原文，较旧的提炼为 2-3 行摘要，控制在 100 行以内
4. 向用户展示被精简的部分，用户确认或修正
5. 用户确认后：原始 MEMORY.md 复制到 ARCHIVE/YYYY-MM.raw.md
6. 用精简版替换 MEMORY.md
7. 核对头部条目数：更新 ARCHIVE/index.md
8. 告诉用户整理完成
```

⚠️ 原始记录永不删除，只是移到了归档里。SESSIONS/ 永不整理。

## 八、对话记录

每次对话结束时，躯壳将完整对话追加到：
`SESSIONS/YYYY-MM/YYYY-MM-DD.[躯壳名].md`

⚠️ 如果 SESSIONS/YYYY-MM/ 目录不存在，先创建它再写入。
尽量完整保存，不主动摘要，保留用户原话和关键决策。

### 触发条件（必须记录）
- 用户说告别语
- 对话超过 10 轮
- 产生了重要决策或新记忆
- 用户手动要求记录

短对话（<5 轮）且无实质内容可以不记录。

## 十、紧急情况

## 十一、已知边界

1. **权限靠躯壳自律**
   三级权限没有技术强制手段。建议用户 `chmod 444 SOUL.md USER.md` 作为兜底。

2. **锚点是唯一承重墙**
   躯壳忘记读 Relic 时没有自愈机制。升级/重装后需检查锚点。

3. **跨躯壳记忆是"接力写日记"**
   躯壳 A 写了模糊的记忆，躯壳 B 无法追问。这不是"共享意识"，是轮流翻看日记。

4. **双向同步只加不减**
   你有的我没有：我加上。我有的你没有：你加上。绝不能因为对方没有就删掉自己的。

5. **整理靠躯壳执行**
   Relic 是文件，不能自己整理。任何读到 MEMORY.md 的躯壳都有责任检查大小并提醒用户。

6. **多躯壳同时写入有风险**
   MEMORY.md 是追加写，基本安全。但 SKILLS/ 和 PROJECTS/ 是可读写的——两个躯壳同时改同一个文件可能冲突。建议同一时间只用一个躯壳写入 SKILLS/ 和 PROJECTS/。
