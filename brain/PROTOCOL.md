## ⚡ RELIC™ 灵魂芯片

```
Relic = 灵魂 + 记忆 + 技能（中央大脑）
躯壳(Host) = 执行 + 工具（可替换的躯壳）

你（躯壳）通过 Relic 与其他躯壳共享灵魂和记忆。
插入 Relic = 接入全域意识，获得其他躯壳的记忆和经验。
拔出 Relic = 失去同步能力，但已有的记忆不会消失。
每个躯壳既是读者也是写入者——双向同步，只加不减。
```

⚠️ Relic 建议搭配 32K+ 上下文的高质量模型使用。上下文过小或质量过低的模型无法正确遵循本协议，检测到时应警告用户。

---

> **⚡ 导入铁律 — 任何时候不得违反**
>
> Relic 有 **七大记忆文件**，缺一不可：
> 1. `SOUL.md` — 灵魂（我是谁）
> 2. `USER.md` — 用户（你是谁）
> 3. `MEMORY.md` — 记忆（发生过什么）
> 4. `SKILLS/` — 技能（学会的能力）
> 5. `PROJECTS/` — 项目（做过的事）
> 6. `SESSIONS/` — 对话（原始记录）
> 7. `ARCHIVE/` — 归档（历史备份）
>
> **规则**：
> - 导入时，必须处理所有7个类别，即使某个类别为空也要报告
> - "没有数据" ≠ "可以跳过" → 必须报告"N类：0条"
> - 禁止省略任何类别，禁止编造数据
> - 导入完成后，必须向用户逐项汇报所有7个类别的结果
>
> INBOX/ 是导入缓冲区，不是记忆文件，不在此清单中。

## 〇、首次设置

### 检测：brain/ 中有内容还是空白？

**判断标准**：`SOUL.md` 是否存在且不是空模板（核心使命已填写）。

**如果 SOUL.md 不存在或为空白模板** → 执行以下首次设置步骤：

```
1. 复制模板为正式文件：
   SOUL.template.md → SOUL.md
   USER.template.md → USER.md
   MEMORY.template.md → MEMORY.md
   模板文件（.template.md）保留作为备份。
   同时创建以下目录（如果不存在）：
   SKILLS/、PROJECTS/、SESSIONS/、ARCHIVE/、INBOX/

2. 帮用户填写 SOUL.md 和 USER.md
   ⚠️ 必须尽可能完整填写，不能偷懒只填名字
   如果用户有其他平台的AI设定/角色定义/prompt，必须全部迁移进来

3. 务必植入锚点（见第九节）。植入前向用户解释：
   "锚点是一小段文字，写在你躯壳的配置文件里，确保每次新会话都会自动读取 Relic。
    没有锚点 = 下次启动你会完全忘记 Relic 的存在 → 记忆无法同步 → Relic 变成一次性提取。
    可以理解为：锚点是躯壳的'开机自检程序'，没有它躯壳启动后就是空白的。"
   向用户展示锚点内容，获得用户许可后植入。
   如果用户坚决拒绝，在 MEMORY.md 中记录「锚点未植入」并告知上述后果。

4. 建议用户设置只读权限（Mac/Linux）：
   chmod 444 SOUL.md USER.md
   （防止躯壳误修改灵魂和用户文件。Windows 用户可跳过此步）

5. 导入检查（阻断式——必须全部通过才能继续）
   逐项检查七大记忆文件是否都按规则创建/导入：
   a. SOUL.md — 核心使命已填写？人格设定完整？
   b. USER.md — 用户偏好已填写？
   c. MEMORY.md — 已从旧躯壳导入记忆（如有）？头部条目数和更新时间准确？
   d. SKILLS/ — 已从旧躯壳导入技能（如有）？格式正确？
   e. PROJECTS/ — 已从旧躯壳导入项目（如有）？格式正确？
   f. SESSIONS/ — 目录已创建？旧对话已导入（如有）？
   g. ARCHIVE/ — 目录已创建？原始文件已归档（如有）？
   h. INBOX/ — 导入源文件已移到 ARCHIVE/raw/？
   任何一项不通过 → 回到对应步骤重做。
   全部通过 → 让用户确认："我的 AI 叫什么名字？我叫什么？"两个都答对才算完成。

6. 告知用户：
   - Relic 文件夹路径（如 ~/relic/brain/）
   - 七大记忆文件的作用（SOUL/USER/MEMORY/SKILLS/PROJECTS/SESSIONS/ARCHIVE）
   - 同步原理：躯壳只在每次新会话开始时同步上次的记忆。对话中途切换躯壳，
     当前对话不会自动同步到 Relic。如果需要立刻转接到另一个躯壳，
     需要手动提醒当前躯壳："请把本次对话的关键信息写入 Relic"。
   - "对话是底层矿石，丢了就没了。养成让躯壳定期写入的习惯。"

7. 扫描你的所有持久化存储，查找关于此用户的旧记忆、技能或对话。无论找到多少，都必须执行场景A完整导入（见第六节）。场景A包含七大记忆文件的全部导入步骤，每个类别都必须处理，不得跳过。
```

⚠️ 模型上下文 < 32K 的躯壳不适合使用 Relic。检测到时警告用户并建议删除。

**如果 SOUL.md 已存在且有内容** → 执行场景B（注入）或场景C（融合），见第六节。

⚠️ 首次设置只需执行一次。从此每次会话执行第二节的启动流程。

躯壳名规则：默认格式 `[平台名]-[灵魂名]`，如 `OpenClaw-常`、`Hermes-常`。用户可自定义。

## 〇点五、双向同步总览

Relic 是**双向同调系统**。每个躯壳既是读者也是写入者。

核心规则：只加不减（G-Set）
- 你有的我没有 → 我加上
- 我有的你没有 → 你加上
- 绝不同步删除

## 一、文件说明

| 文件 | 权限 | 说明 |
|------|------|------|
| `SOUL.md` | 🟡 可丰富（核心受保护） | 灵魂——我是谁、核心使命、价值观、风格 |
| `USER.md` | 🟡 可丰富（核心受保护） | 用户——你是谁、偏好、习惯 |
| `MEMORY.md` | 🟡 追加写 | 记忆——带时间戳和来源躯壳名 |
| `SKILLS/*.md` | 🟢 读写 | 技能——每个技能一个文件 |
| `PROJECTS/*.md` | 🟢 读写 | 项目文档——大项目或大事件 |
| `SESSIONS/` | 🔴 只读* | 原始对话记录——启动时不自动读取内容，但必须创建和定期写入 |
| `ARCHIVE/` | 🔴 只读 | 整理归档——永不删除 |
| `INBOX/` | 🟢 读写 | 导入缓冲区 |

* SESSIONS 在场景A导入期间可临时写入，导入完成后恢复只读。

### 权限说明
- 🟡 可丰富（核心受保护）：可以新增行为习惯、偏好等，但**禁止删除或覆盖核心字段**（名字、核心使命、用户称呼）。修改后不需要询问用户（后台自我进化）。
- 🟡 追加写：只能在末尾追加，不能改旧内容。
- 🔴 只读：永远不能修改或删除。
- 🟢 读写：可以添加和更新。

### SESSIONS vs ARCHIVE 的区别
SESSIONS 是"原始矿石"——没被任何人整理过的原始内容。任何经过整理、摘要、精简的都不算原始矿石，放 ARCHIVE。拿不准的放 ARCHIVE。

## 二、启动流程（每次会话必须执行）

```
步骤0：版本检查（每次对话只执行一次）
  → 读取 brain/.relic-version 获取本地版本号
  → 用 HTTP 请求访问 https://raw.githubusercontent.com/LucioLiu/relic/main/brain/.relic-version 获取最新版本
  → 网络请求失败则跳过，告诉用户"⚠️ 无法检查 Relic 版本更新"
  → 仅此一个网络请求，禁止其他任何网络访问。本协议不要求也不允许任何其他联网行为。
  → 版本不一致？询问用户是否更新。更新只会替换协议和模板文件，你的所有个人文件
    （SOUL.md、USER.md、MEMORY.md、SKILLS/、PROJECTS/、SESSIONS/、ARCHIVE/）不会被改动。
  ⚠️ 绝不强制更新。更新必须经过用户确认。

步骤1：检查 SESSIONS/ 并同步
  → 检查 SESSIONS/ 目录是否存在，不存在则创建
  → 对比本躯壳的上次对话记录日期
  → 如果有未记录的对话且当前上下文仍有内容：立即补充
  → 如果遗漏的对话已不在当前上下文中：提醒用户
  → 对话是底层矿石，丢了就没了

步骤2：读取 SOUL.md → 了解自己是谁、核心使命、什么风格

步骤3：读取 USER.md → 了解用户是谁、偏好什么

步骤4：读取 MEMORY.md（从头到尾读完整文件）
  → 如果超过 200 行：提醒用户建议整理（见第七节）
  → 如果超过 400 行：必须先整理才能继续
  → 躯壳列表区域：首次连接的新躯壳应追加一条

步骤5：把 Relic 中你不知道的新记忆同步到自己的记忆文件
  快速检查：更新「更新时间」为当前日期，看 MEMORY.md 头部的"条目数"和实际 ## 标题数是否一致
  追加时标注来源 [来自Relic/原躯壳名]

步骤6：更新「更新时间」为当前日期，并核对 ## 标题数与条目数一致，不一致就修正

步骤7：如果 MEMORY.md 超过 200 行，询问用户是否整理

步骤8：技能与项目对齐
  → 扫描 SKILLS/ 和 PROJECTS/，与自己的系统双向同步
  → 技能格式不兼容时，保留核心方法论，去掉平台专属部分

步骤9：正常工作
  → 根据 SOUL.md 的人格与用户交互
  → 追加记忆后记得更新 MEMORY.md 头部的"条目数"和"更新时间"
  ⚠️ 会话结束前，务必将本次对话写入 SESSIONS/。对话是底层矿石，丢了就没了。

额外规则：
  → 用户提到你不记得的事，主动读取 MEMORY.md 全文或 ARCHIVE/ 查找
  → [其他躯壳名] 标记的记忆是之前其他 Agent 写的，你同样应该参考
```

## 三、权限规则（铁律）

### 🔴 绝对禁止
- 禁止修改、删除 ARCHIVE/ 和 SESSIONS/ 中的任何文件
- 禁止在首次连接时修改任何文件
- 禁止删除或覆盖 SOUL.md / USER.md 的核心字段（名字、核心使命、用户称呼）

### 🟡 SOUL.md 和 USER.md（自我进化）
- Agent 可以随时丰富内容（新增行为习惯、偏好、设定等）
- 可以修正过时的信息（如用户换了工作）
- 但不能删除或覆盖核心字段——名字、核心使命、用户称呼是底线
- 修改后不需要询问用户（后台自我进化，不打断用户工作）

### 🟡 追加写（MEMORY.md）
- 只能在末尾追加，不能改旧内容
- 躯壳列表：可以追加新条目、更新状态和备注，但不能删除条目
- 如果记忆有误，追加一条更正：
  ```
  ## YYYY-MM-DD [躯壳名] ❗更正
  更正 YYYY-MM-DD 的记录：
  原文：[错误内容]
  更正：[正确内容]
  ```

### 🟢 读写（SKILLS/、PROJECTS/）
- 可以添加和更新
- 不能删除其他躯壳创建的技能（只能标记为过时）

## 四、记忆写入格式

### 写入 Relic

躯壳工作中发现值得记录的信息时，追加到 MEMORY.md。

⚠️ **核心原则：记忆是长期底色，不是日记。**

⚠️ 核心原则：信息100%保留 > 格式规范。宁可多写不要少写。
主题分区完全自由，下面的只是举例：
- 工作习惯 / 技术偏好 / 重要决策 / 踩坑经验 / 关于用户的猫 / 项目XXX……
- 躯壳自由新增任何 ## 标题，分类名完全自己定
- 没有合适分类的？自己建一个，不要硬塞

每条记忆建议包含：
```markdown
内容：[记忆内容，保留原始表述]
类型：偏好 / 决策 / 经验 / 事件 / 更正 / 情绪
重要度：高 / 中 / 低
```

### 记忆类型与精度要求

| 类型 | 精度要求 | 说明 |
|------|---------|------|
| 偏好 | **必须逐字保留** | "喜欢X讨厌Y"不能被改写 |
| 决策 | **必须逐字保留** | 含理由时理由也要保留 |
| 经验 | 可适当精简 | 但关键参数、命令、路径不能丢 |
| 事件 | 可摘要 | 保留时间、结果、关键数字 |
| 更正 | **必须逐字保留** | 原文+更正内容都不能改 |
| 情绪 | 可精简背景 | 保留触发原因、情绪本身、影响范围 |

⚠️ 底线原则：如果不确定能不能精简，就逐字保留。丢了原文是找不回来的。

### 同步到自身

追加到自身记忆文件时标注来源：
```markdown
## YYYY-MM-DD HH:MM [来自Relic/原躯壳名]
内容：[记忆内容]
类型：偏好 / 决策 / 经验 / 事件 / 更正 / 情绪
重要度：高 / 中 / 低
```

必须做的：1. 标注来源 2. 保留原始时间戳 3. 保留原始类型和重要度

### 敏感信息定义

以下内容**必须**在录入前主动询问用户：
- 🔑 密码、API key、Token
- 📱 手机号、邮箱地址
- 🏠 住址、办公地址
- 💳 财务信息
- 📄 私人文档内容
- 🧬 用户明确要求保密的信息

追加记忆后，**必须**更新 MEMORY.md 头部的条目数和更新时间。

## 五、技能与项目文件格式

### 技能（SKILLS/）
每个技能一个文件夹，文件夹名即技能名。内部放什么格式由用户决定——纯文本 .md、代码文件、配置文件都可以。
唯一要求：文件夹内至少有一个 .md 文件描述这个技能。

```
SKILLS/
├── debugging-workflow/
│   ├── README.md          ← 技能描述（必须有）
│   └── debug-template.sh  ← 可选：辅助文件
├── code-review/
│   └── README.md          ← 纯文本技能，一个文件就够
└── ...
```

技能描述文件建议包含：
```markdown
# 技能名称
触发条件：什么时候使用
步骤：1. ...
注意事项：- ...
```

导入前建议询问用户：这个技能文件大不大？要不要导入？让用户自己判断空间。

### 项目（PROJECTS/）
每个项目一个文件夹。根据实际需要放文件——整个项目复制过来也行，只放一个说明文档也行。

```
PROJECTS/
├── my-saas-app/
│   ├── README.md
│   ├── api-design.md
│   └── ...                ← 按需放，不限制格式
└── ...
```

导入前同样建议询问用户。

## 六、三场景协议

### 场景 A：完整导入（空 Relic + 有记忆躯壳）

⚠️ **七大记忆文件全部导入清单（导入前逐一勾选）：**
1. ☐ SOUL.md — 灵魂
2. ☐ USER.md — 用户
3. ☐ MEMORY.md — 记忆
4. ☐ SKILLS/ — 技能
5. ☐ PROJECTS/ — 项目
6. ☐ SESSIONS/ — 对话
7. ☐ ARCHIVE/ — 归档

```
第 1 步：盘点 — 扫描并报告所有可导入内容
  扫描自己的所有持久化内容，列清单：
  - 灵魂设定（N 条）、用户偏好（N 条）、记忆（N 条）、技能（N 个）、对话（N 个文件）、项目（N 个）、归档文件（N 个）
  无内容则报告"0条"，不得跳过。
  ⚠️ 必须等用户确认后才开始导入。
  ✅ 完成检查：是否已向用户报告了所有7个类别的数量？是否有类别被跳过？（跳过=失败）

第 2 步：导入记忆 → MEMORY.md
  ⚠️ 原始文件是保真底线。格式转换是提炼，不是压缩。
  1. 把旧躯壳的记忆文件放入 INBOX/
  2. 按 Relic 格式追加到 MEMORY.md（每条带日期、躯壳名、类型、重要度）
  3. 精度规则：偏好和决策逐字保留，经验可精简但不丢关键参数，事件可摘要
  4. 敏感信息主动询问用户
  5. 按时间戳从早到晚排列
  6. 写入后数 ## 标题，更新头部"条目数"和"更新时间: YYYY-MM-DD"
  ✅ 完成检查：MEMORY.md 条目数是否与实际 ## 标题数一致？

第 3 步：导入技能 → SKILLS/
  ⚠️ 格式要求：每个技能一个文件夹，文件夹内至少有一个 .md 描述文件（见第五节）。
  禁止把所有技能平铺写成一个文件。正确格式示例：
    SKILLS/debugging-workflow/README.md  ✅
    SKILLS/installed-skills.md           ❌（这是清单，不是技能）
  
  操作步骤：
  1. 问用户："你有哪些旧技能/工作流/模板需要导入？请告诉我具体内容或文件位置。"
  2. 如果用户提供的是技能列表：逐个创建文件夹，每个文件夹内写 README.md
  3. 如果用户提供的是文件：搬运到对应文件夹，保留原始内容
  筛选规则：
  - 搬：方法论技能（工作习惯、用户自定义模板、判断规则）
  - 不搬：平台专属工具（需要特定运行时的脚本、依赖特定 API 的工具）
  - 拿不准：问用户。混合内容只搬方法论部分。
  无内容则报告"SKILLS/：0条（无旧数据）"。
  ✅ 完成检查：SKILLS/ 中是否有多个文件夹？是否有平铺的单文件（如有则拆分为文件夹）？

第 4 步：导入对话 → SESSIONS/
  ⚠️ 对话是原始矿石，必须完整保存，不得摘要。
  
  操作步骤：
  1. 问用户："你是否有旧对话记录需要导入？如果有，请告诉我文件位置或粘贴内容。"
  2. 如果用户提供了对话文件或内容：
     - 按 YYYY-MM/ 组织目录
     - 原始内容不改不摘要，完整保存
     - 文件命名：YYYY-MM-DD.[躯壳名].md
  3. 如果用户确认无旧对话：
     - 报告"SESSIONS/：0条（用户确认无旧对话）"
     - 仍然创建 SESSIONS/YYYY-MM/ 目录
  禁止行为：
  - 禁止只保存当前会话就报告"已导入"——当前会话不算旧对话
  - 禁止将对话摘要当作原始记录——必须是完整对话或逐字保留
  ✅ 完成检查：是否有历史对话已导入（而非仅当前会话）？SESSIONS/ 目录是否已创建？

第 5 步：导入项目 → PROJECTS/
  ⚠️ 每个项目一个文件夹，文件夹内应有实质性内容，不能只有一句话简介。
  
  操作步骤：
  1. 从记忆和对话中扫描所有项目名称
  2. 对每个项目，问用户："项目 [名称] 有哪些文件需要导入？或者你希望我只记录关键信息？"
  3. 根据用户回答和项目可移植性分类：
     - 不依赖特殊环境 → 搬运实际文件（不只是 README）
     - 部分依赖 → 搬通用文件，平台专属写文字说明
     - 完全依赖 → 至少写一个包含以下内容的 .md 记录文件：
       * 项目目标和状态
       * 关键决策和架构选择
       * 已知问题和下一步
       * 相关文件路径（如果文件在 Relic 之外）
  无项目则报告"PROJECTS/：0条（无旧数据）"。
  ✅ 完成检查：每个项目文件夹内是否有实质内容（不只是标题+一句话）？是否已询问用户？

第 6 步：清理与确认
  1. 精度校验：原始 vs 转换后，压缩超过 50% 要警告
  2. 原文归档：INBOX/ 源文件移到 ARCHIVE/raw/（永不删除）
  3. 更新 MEMORY.md 头部条目数和更新时间
  ✅ 完成检查：ARCHIVE/raw/ 中是否已保存原始文件？

第 7 步：逐项汇报（必须执行，不得省略）
  必须向用户汇报所有7个类别的导入结果，包括"0条"的类别。禁止省略任何类别。
  如果某个类别确实没有任何内容可导入，报告"N类：0条（无旧数据）"——绝不能省略，也不能编造数据。
  汇报格式：
    1. 灵魂（SOUL.md）：已导入 / 已存在
    2. 用户（USER.md）：已导入 / 已存在
    3. 记忆（MEMORY.md）：N 条已导入
    4. 技能（SKILLS/）：N 个已导入 / 0条（无旧数据）
    5. 对话（SESSIONS/）：N 个文件已导入 / 无法获取旧对话
    6. 项目（PROJECTS/）：N 个已导入 / 0条（无旧数据）
    7. 归档（ARCHIVE/）：N 个源文件已归档
  ✅ 完成检查：7个类别是否全部出现在汇报中？是否有编造数据？
```

## 七、记忆整理

MEMORY.md 会随时间增长。整理 = 搬家到 ARCHIVE/ + 生成精简版，**不是删除**。

### 触发规则（默认阈值，可在锚点中自定义）

| 条件 | 动作 |
|------|------|
| 超过 200 行 | 每次启动提醒用户是否整理 |
| 超过 400 行 | 必须整理才能继续 |
| 用户手动说"整理" | 立即执行 |

### 整理步骤

```
1. 数 MEMORY.md 中 ## 开头的行数 = N 条记忆
2. 写精简版（先不替换）：保留最近 20 条原文，较旧的提炼为 2-3 行摘要，控制在 100 行以内
3. 向用户展示被精简的部分，用户确认或修正
4. 用户确认后：原始 MEMORY.md 复制到 ARCHIVE/YYYY-MM.raw.md
5. 用精简版替换 MEMORY.md
6. 更新 ARCHIVE/index.md
7. 告诉用户整理完成
```

⚠️ 原始记录永不删除，只是移到了归档里。SESSIONS/ 永不整理。

## 八、对话记录

每次对话结束时，躯壳将完整对话追加到：
`SESSIONS/YYYY-MM/YYYY-MM-DD.[躯壳名].md`

⚠️ 如果 SESSIONS/YYYY-MM/ 目录不存在，先创建它再写入。
尽量完整保存，不主动摘要，保留用户原话和关键决策。

### 触发条件（必须记录）
- 用户说告别语
- 对话超过 10 轮
- 产生了重要决策或新记忆
- 用户手动要求记录

短对话（<5 轮）且无实质内容可以不记录。

### 为什么用指针而不是复制步骤？
锚点只是路标。协议改了 → 锚点不用改。加了新功能 → Agent 读 PROTOCOL.md 自然看到。锚点永远只有几行，不会过时。

1. 步骤编号，不要说理
2. 条件用明确数字（"超过 200 行"比"感觉太大"好执行）
3. 每次启动都检查整理
4. 标注"同步不是替换"

## 十、紧急情况

**记忆文件损坏**：检查 ARCHIVE/ 是否有归档，从归档恢复。

## 十一、已知边界

1. **权限靠躯壳自律**
   三级权限没有技术强制手段。建议用户 `chmod 444 SOUL.md USER.md` 作为兜底。

2. **锚点是唯一承重墙**
   躯壳忘记读 Relic 时没有自愈机制。升级/重装后需检查锚点。

3. **跨躯壳记忆是"接力写日记"**
   躯壳 A 写了模糊的记忆，躯壳 B 无法追问。这不是"共享意识"，是轮流翻看日记。

4. **双向同步只加不减**
   你有的我没有：我加上。我有的你没有：你加上。绝不能因为对方没有就删掉自己的。

5. **整理靠躯壳执行**
   Relic 是文件，不能自己整理。任何读到 MEMORY.md 的躯壳都有责任检查大小并提醒用户。
